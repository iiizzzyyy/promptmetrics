# PromptMetrics v1.0.1: From Prototype to Production

> Every prompt registry works beautifully in a local demo. It is when the security questionnaire lands on your desk, when a prompt change silently degrades production output, and when two teams need isolated workspaces that you discover whether a tool is a toy or infrastructure.

PromptMetrics v1.0.1 is our answer to the question we have heard most from early users: **"Can we actually run this in production?"**

PromptMetrics is a lightweight, self-hosted prompt registry that treats prompt content as code, not data, giving you versioning, metadata logging, and observability without vendor lock-in or monthly SaaS fees. Today we are shipping version 1.0.1, and this is the release that moves PromptMetrics from a handy single-node registry to a production-grade platform you can run beside the rest of your infrastructure.

---

## The Theme: From Prototype to Production

Version 1.0.1 is all about **trust**. Early adopters proved the core concept — you can version prompts, sync them from GitHub, and trace every LLM call — but production demands more. It demands defense in depth, horizontal scalability, and tenant isolation.

This release follows a deliberate arc: **first earn trust through security hardening, then earn adoption through enterprise readiness.** If v1.0.0 proved prompts deserve version control, v1.0.1 proves that version control deserves production infrastructure.

---

## Security Hardening for Production Deployments

Self-hosting gives you control, but control requires vigilance. In 1.0.1, we systematically closed every gap that could turn a prompt registry into an attack vector.

- **Path traversal prevention** — The `FilesystemDriver` and `GithubDriver` now sanitize all user-supplied paths before they reach the operating system. A malformed relative path can no longer climb out of its designated directory.
- **Input validation** — The `EvaluationController` enforces strict Joi schemas on every payload, so malformed requests bounce before they touch your business logic.
- **Rate limiter race condition fixed** — We wrapped the SQLite counter increment inside atomic `db.transaction()` blocks. Before this change, rapid concurrent requests could slip past the limit; now the counter is accurate even under burst load.
- **Webhook security hardened** — The GitHub webhook handler now fails closed. If `GITHUB_WEBHOOK_SECRET` is missing, the endpoint returns an immediate rejection. We removed the old `GITHUB_TOKEN` fallback because a webhook endpoint that accepts unsigned traffic is not a webhook endpoint at all.
- **Sensitive logging removed** — We stripped a `console.log(JSON.stringify(...))` statement from `LogController` that was accidentally leaking LLM metadata to stdout.
- **SQL injection prevention** — Dynamic `tableName` and `columnName` values are now validated against `/^[a-z_][a-z0-9_]*$/i` before SQL interpolation. Your database schema stays exactly where it belongs: under your control.

These fixes are unglamorous, but they are exactly what unblock an engineering team from deploying.

## Feature Highlights

### Evaluation Framework

Before 1.0.1, measuring prompt quality meant shipping to production and waiting for user complaints. Now you create, score, and manage prompt evaluations through a dedicated REST API. You can attach numeric scores, labels, and metadata to any prompt version, turning subjective "this feels worse" into reproducible metrics.

```bash
curl -X POST http://localhost:3000/v1/evaluations \
  -H "Content-Type: application/json" \
  -H "X-API-Key: $API_KEY" \
  -d '{
    "prompt_name": "summary_v3",
    "version": "1.0.0",
    "score": 0.94,
    "metadata": {
      "model": "gpt-4.1",
      "dataset": "bbc_news_1k"
    }
  }'
```

You now treat prompt regression like code regression: you catch it before it reaches users.

### Web UI Dashboard

Managing prompts through `curl` and raw JSON is fine for automation, but operators need visibility. The new Next.js dashboard gives you a unified interface for browsing prompts, inspecting logs and traces, reviewing evaluation runs, managing labels, and configuring settings. Your product managers can audit prompt versions without filing tickets to engineering, and your SREs can inspect traces without parsing log files.

### Python SDK

Hand-rolling HTTP clients for every internal tool is tedious and error-prone. The new Python SDK, located in `clients/python/`, exposes the full PromptMetrics API surface with native methods and types. You install it, authenticate once, and start versioning prompts from your training pipelines or Jupyter notebooks. It turns integration from an afternoon of boilerplate into a five-minute import.

### Production-Ready Infrastructure

Running a single SQLite instance on your laptop is great for a proof of concept; running production traffic on it is not. In 1.0.1, you swap `DATABASE_URL` to a PostgreSQL connection string and the application switches backends automatically. Point `REDIS_URL` to a Redis cluster and you get LRU caching for prompt lookups and distributed sliding-window rate limiting with `Retry-After` headers. Need durable object storage? Set the `DRIVER` environment variable to `s3` and store prompt artifacts on any S3-compatible service.

For teams managing multiple products or customers, the new multi-tenancy layer isolates workspaces via the `X-Workspace-Id` header, so one tenant's prompts never leak into another's namespace. Together, these changes mean you can deploy PromptMetrics behind your existing load balancers and identity providers without architectural gymnastics.

## Moving the Vision Forward

PromptMetrics was born from a simple idea: **prompts deserve the same rigor as the code that calls them.** Version 1.0.1 advances that vision by closing the gap between "works on my machine" and "runs in production."

Our vision has always been to be the "Git for prompts" — versioned, diffable, portable content. But code without CI, without access control, and without observability is just text in a file. v1.0.1 adds the platform layer that makes the vision viable for teams shipping real products.

You now have:
- Circuit breakers on GitHub API calls
- Per-API-key rate limiting with expiration dates
- An async audit-log queue for compliance
- A migration system powered by Umzug
- OpenAPI documentation served live at `/docs`
- The tooling to treat prompt changes with the same discipline you apply to schema migrations and feature flags

This is what "Git for prompts" looks like at scale: versioned, observable, secure, and entirely under your own infrastructure.

## Get Started Today

Ready to stop treating prompts as second-class artifacts?

```bash
npm install -g promptmetrics
promptmetrics-server
```

Or pull the Docker image, set your `DATABASE_URL`, and open the dashboard. If you are already running an earlier version, restart with the new environment variables and review the updated configuration guide.

Star the project on [GitHub](https://github.com/iiizzzyyy/promptmetrics) to follow the roadmap, and open an issue if you hit a snag. We built PromptMetrics so you can own your prompt layer — v1.0.1 makes that ownership safe, scalable, and complete.
